### Задание 1. СУБД

### Кейс
Крупная строительная компания, которая также занимается проектированием и девелопментом, решила создать 
правильную архитектуру для работы с данными. Ниже представлены задачи, которые необходимо решить для
каждой предметной области. 

Какие типы СУБД, на ваш взгляд, лучше всего подойдут для решения этих задач и почему? 
 
1.1. Бюджетирование проектов с дальнейшим формированием финансовых аналитических отчётов и прогнозирования рисков.
СУБД должна гарантировать целостность и чёткую структуру данных.

1.1.* Хеширование стало занимать длительно время, какое API можно использовать для ускорения работы? 

1.2. Под каждый девелоперский проект создаётся отдельный лендинг, и все данные по лидам стекаются в CRM к 
маркетологам и менеджерам по продажам. Какой тип СУБД лучше использовать для лендингов и для CRM? 
СУБД должны быть гибкими и быстрыми.

1.2.* Можно ли эту задачу закрыть одной СУБД? И если да, то какой именно СУБД и какой реализацией?

1.3. Отдел контроля качества решил создать базу по корпоративным нормам и правилам, обучающему материалу 
и так далее, сформированную согласно структуре компании. СУБД должна иметь простую и понятную структуру.

1.3.* Можно ли под эту задачу использовать уже существующую СУБД из задач выше и если да, то как лучше это 
реализовать?

1.4. Департамент логистики нуждается в решении задач по быстрому формированию маршрутов доставки материалов 
по объектам и распределению курьеров по маршрутам с доставкой документов. СУБД должна уметь быстро работать
со связями.

1.4.* Можно ли к этой СУБД подключить отдел закупок или для них лучше сформировать свою СУБД в связке с СУБД 
логистов?

1.5.* Можно ли все перечисленные выше задачи решить, используя одну СУБД? Если да, то какую именно?

*Приведите ответ в свободной форме.*

### Решение

1.1.  Для жесткой и целостной структуры данных подходят реляционные СУБД, например PostgreSQL обладает полной ACID-совместимостью, обеспечивающей целостность данных в сложных транзакциях. ​
1.1.* Если с точки зрения ИБ допустимо, то можно понизить алгоритмы например до sha-256 вместо 512. Вынести хранение уже вычисленных хешей в кэш во внешние сервисы например, Redis, Memcached или внутреннем кэше СУБД, чтобы не пересчитывать их каждый раз при запросе.
1.2.  Для лендингов подойдет NoSQL вроде MongoDB за гибкость и скорость вставок лидов; для CRM — PostgreSQL или Airtable для быстрых запросов и интеграций с маркетингом и продажами.
1.2.* Можно использовать под эту задачу только MongoDB или PostgreSQL с поддержкой JSONB.
1.3.  Реляционная СУБД вроде MariaDB или PostgreSQL с простой табличной структурой подойдет для иерархических данных компании.
1.3.* Да, интегрировав в PostgreSQL из 1.1 как отдельную схему с таблицами для категорий и материалов.
1.4.  Графовая СУБД вроде Neo4j или Memgraph excels в связях для маршрутов, курьеров и объектов благодаря traversal-алгоритмам. Или реляционная СУБД с оптимизацией запросов и GIS-расширениями, например PostgreSQL+PostGIS.
1.4.* Лучше оставить в отдельной реляционной СУБД, так как это финансовые операции и настроить интеграцию.
1.5.* Можно попробовать использовать PostgreSQl с дополнительными модулями (JSONB, PostGIS) для всех текущих задач.

### Задание 2. Транзакции

2.1. Пользователь пополняет баланс счёта телефона, распишите пошагово, какие действия должны произойти для того, чтобы 
транзакция завершилась успешно. Ориентируйтесь на шесть действий.

2.1.* Какие действия должны произойти, если пополнение счёта телефона происходило бы через автоплатёж?

*Приведите ответ в свободной форме.*

### Решение

1) Инициация и проверка (Begin Transaction): Система проверяет корректность введенного номера и доступность платежного шлюза.
2) Блокировка суммы (Hold): В банке плательщика резервируется сумма, создается запись о транзакции в статусе «в процессе».
3)Списание средств: Подтверждение банком успешного списания; данные передаются через платежный шлюз провайдеру связи.
4)Обновление баланса в БД+запись лога транзакции: СУБД биллинга оператора связи выполняет операцию UPDATE, увеличивая остаток на счету абонента.
5)Фиксация транзакции (Commit): Сохранение изменений во всех связанных базах данных, перевод статуса платежа в «успешно».
6)Уведомление: Отправка подтверждения пользователю (push/SMS) и формирование чека.

При автоплатеже добавляются шаги триггерной проверки:
1)Система мониторинга баланса в реальном времени (или по расписанию) обнаруживает достижение «порога» (например, менее 50 руб.).
2)Генерация автоматического запроса в банк на основе ранее сохраненного и подтвержденного токена (рекуррентный платеж) без участия пользователя.
3)Если банк отклоняет транзакцию (нет средств), система должна выполнить логику Retry (повтор через время) или уведомить пользователя об отключении услуги.

### Задание 3. SQL vs NoSQL

3.1. Напишите пять преимуществ SQL-систем по отношению к NoSQL. 

3.1.* Какие, на ваш взгляд, преимущества у NewSQL систем перед SQL и NoSQL.

*Приведите ответ в свободной форме.*

### Решение

Пять преимуществ SQL над NoSQL:
1)Соответствие ACID: Гарантированная надежность, целостность и согласованность данных.
2)Стандартизация: Использование единого языка SQL упрощает поиск специалистов и миграцию между системами (PostgreSQL, MySQL, Oracle). Большое количество источников технической документации.
3)Сложные связи (Joins): Возможность эффективно объединять данные из десятков таблиц одним запросом.
4)Отсутствие дублирования: Нормализация данных экономит место и исключает аномалии при обновлении информации.
5)Широкий фукционал: Наличие продвинутых инструментов администрирования, безопасности и глубокой аналитики.

Преимущества NewSQL:*
1)Масштабируемость NoSQL: Они могут горизонтально расширяться на сотни серверов (в отличие от классических SQL).
2)Надежность SQL: При этом они полностью поддерживают ACID и стандартный SQL.
3)В отличие от старых БД, NewSQL сами распределяют данные по узлам, избавляя разработчиков от ручного управления кластером.

### Задание 4. Кластеры

Необходимо производить большое количество вычислений при работе с огромным количеством данных, под эту задачу 
выделено 1000 машин. 

На основе какого критерия будете выбирать тип СУБД и какая модель *распределённых вычислений* 
здесь справится лучше всего и почему?

*Приведите ответ в свободной форме.*

### Решение

Критерий выбора СУБД:
Главными критериями будут масштабируемость (Scalability) и отказоустойчивость (Fault Tolerance). СУБД должна эффективно работать в архитектуре Shared Nothing, где каждый из 1000 узлов независим, чтобы избежать «узкого горлышка» в виде центрального сервера.
Модель распределённых вычислений:
Лучше всего справится модель MapReduce.
1)Огромный массив данных разбивается на мелкие части и распределяется по 1000 машинам для параллельной обработки.
2)Вычисления происходят там же, где лежат данные, что минимизирует пересылку огромных объемов информации по сети.
3)Результаты с каждой машины собираются и объединяются в итоговый ответ.
4)Если 5-10 машин из 1000 выйдут из строя, система просто переназначит их задачи на другие узлы без прерывания общего процесса.
